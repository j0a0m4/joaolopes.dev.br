Design: Section 1 of 3 — Hamburger nav

Approach B, a11y-first. Every decision here has a reason rooted in WCAG 2.2 or ARIA APG patterns.

HTML structure

layout.clj change — nav becomes
;; <button> not <div> — native keyboard + screen reader target
[:button.nav-toggle
  {:aria-expanded "false"
   :aria-controls "nav-menu"
   :aria-label "Menu"}
  [:span.hamburger-icon {:aria-hidden "true"} "☰"]]

;; nav-links gets an id + hidden state
[:span#nav-menu.nav-links {:hidden ""}
  ...]

A11y contract

  • aria-expanded toggles true/false on click — screen readers announce "Menu, collapsed" / "Menu, expanded"
  • aria-controls="nav-menu" links button to the list it controls — VoiceOver/TalkBack can jump directly to the menu
  • hidden attribute (not display:none via CSS) — when hidden, the nav items are truly removed from the accessibility tree, not just visually hidden
  • Escape key closes menu, returns focus to button — standard keyboard pattern per ARIA APG disclosure nav
  • Touch target ≥44×44px — hamburger button gets min-height/width: 44px (WCAG 2.5.5)
  • No focus trap — a disclosure nav (not a modal) lets Tab flow naturally through the items then out
