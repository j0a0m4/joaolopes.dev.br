Three implementation approaches

You've chosen: hamburger nav + collapsible TOC. Now: how much JS should this involve?

A

Zero JS — <details> for everything

HTML5 <details>/<summary> handles both the nav dropdown and TOC collapse natively — no JavaScript at all. Browser renders open/close for free. Fully accessible. Minor trade-off: can't animate the open/close smoothly, and the nav toggle button looks like a native disclosure widget unless you style it away.

Pros

  • Zero JS
  • Accessible by default
  • Works without JS enabled

Cons

  • Harder to style the ☰ button
  • No smooth animation
B ★ recommended

Minimal JS — ~20 lines inline

Nav hamburger toggle via a tiny classList.toggle (same pattern as highlight.js already uses). TOC collapse via <details>/<summary> — no JS needed there. Keeps your Clojure layout clean. Adds one small inline <script> tag to the base layout.

Pros

  • Fully styleable hamburger
  • Smooth CSS transitions
  • TOC still zero-JS
  • ~20 lines total

Cons

  • Nav needs JS enabled
C

Pure CSS hamburger — checkbox hack

Hidden <input type="checkbox"> + <label> drives the nav open/close via CSS sibling selectors. Zero JS, fully styleable. The CSS is more complex than it looks (z-index, positioning, overlay dismiss). Clever but brittle.

Pros

  • Zero JS
  • Fully styleable

Cons

  • Complex CSS
  • Fragile on edge cases
  • Accessibility needs extra work

All approaches include these same CSS fixes

  • Tags: add flex-wrap:wrap — one line
  • Code blocks: right-edge fade gradient to signal overflow
  • Dateline: allow wrapping gracefully with flex-wrap
  • Body padding: slightly more horizontal breathing room on phones
  • Breakpoint target: ≤480px for phones, no changes above that
