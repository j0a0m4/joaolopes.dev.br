Implementation approaches

Experience is settled. Now: how does the pipeline generate the diagram pages?

A ★

Convention-driven — derive everything from filenames

Title: agent-loop.svg → "Agent Loop" (kebab-to-title). Back-link: scan all posts at build time to find which post references each SVG — no author config needed. SVG <title> injected during inlining. Zero new config format.

Pros

  • No config to maintain
  • Works for existing SVGs today
  • Adding a new SVG auto-generates its page

Cons

  • Title quality depends on filename
  • If an SVG appears in multiple posts, need a tie-breaking rule
B

Explicit registry — EDN/YAML config file

A diagrams.edn file maps each SVG to its title, description, and parent post. Authoritative, flexible — can write a custom description for each diagram. Requires updating when adding a new diagram.

Pros

  • Rich metadata per diagram
  • Explicit and predictable

Cons

  • Manual maintenance
  • Existing SVGs need backfill
C

Post frontmatter — declare diagrams in the post

Each post lists its diagrams in YAML frontmatter: diagrams: [{file: agent-loop.svg, title: "Agent Loop"}]. The build derives diagram pages from the posts that declare them. Tight coupling between post and its diagrams.

Pros

  • Diagrams stay with their post context
  • Natural place for diagram metadata

Cons

  • Adds YAML to posts already managed in Obsidian
  • More frontmatter bloat
My recommendation: A. Convention over configuration fits the existing codebase style. Filenames are already meaningful (mcp-architecture, hooks-lifecycle). Scanning posts to find back-links is a build-time operation that costs nothing at runtime. The 5 existing SVGs all have clean filenames that produce good titles.
